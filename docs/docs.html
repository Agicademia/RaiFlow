Installation

RaiFlow requires Python 3.9 or later. Install from PyPI:

pip install raiflow
Optional: For semantic LLM checks (--enable-llm-checks), you also need either Ollama running locally or a Google Gemini / OpenAI API key. The raiflow init wizard will guide you through setup.

Quick Start

Scaffold your project

Run raiflow init in your project root. RaiFlow scans .py files for AI framework imports, infers a risk level, and writes raiflow.yaml and .github/workflows/rai-compliance.yml.

Edit raiflow.yaml

Fill in the paths to your risk assessment, technical documentation, and override endpoints. The generated file is pre-commented to guide you.

Run your first check

Run raiflow check --stage ci to open the live dashboard and see results stream in real-time. Use --no-dashboard for terminal-only output.

Push and let CI gate your pipeline

Every pull request now runs compliance checks. Failures block merges. A signed JSON report is attached as a build artifact.

raiflow.yaml — Manifest Schema

The manifest is the single source of truth for your compliance configuration. raiflow init generates a pre-filled version. Fields marked optional default gracefully — checks that depend on absent fields are reported as skipped rather than failing.

raiflow.yaml 
# Required
system_name: "my-rag-app"
risk_level: "high"          # high | limited | minimal
compliance_framework: "eu_ai_act"

# Article 13 — Transparency
model_metadata:
  name: "llama3"
  version: "3.1"
  disclosure_flag: true
  disclosure_text_path: "docs/disclosure.md"  # optional, for LLM eval

# Article 9 — Risk Management
risk_management:
  assessment_path: "docs/risk_assessment.md"

# Article 14 — Human Oversight
oversight:
  override_endpoints:
    - "/api/override"
    - "/api/halt"

# Article 12 — Logging
logging:
  middleware_active: true

# Article 10 — Data Governance & Bias
data_governance:
  dataset_path: "data/training.csv"       # optional
  protected_attributes: ["gender", "ethnicity"]

# Article 11 — Technical Documentation
technical_documentation:
  path: "docs/tech_doc.md"

# Article 10 — Robustness
robustness:
  red_team_prompts_path: "tests/red_team.txt"  # optional
  toxicity_threshold: 0.7

banned_models: []
Tip: Fields like dataset_path, red_team_prompts_path, and technical_documentation.path are optional. Checks that depend on absent fields are reported as skipped rather than failing.

raiflow init

Scans your project for AI framework imports (LangChain, OpenAI, Ollama, LlamaIndex, Transformers, FastAPI, and more), infers your EU AI Act risk level, and writes two files:

  • raiflow.yaml — pre-filled compliance manifest
  • .github/workflows/rai-compliance.yml — ready-to-use GitHub Actions workflow
raiflow init
raiflow init --directory ./my-ai-project
raiflow init --force   # overwrite existing files

Options

FlagDefaultDescription
--directory.Root directory to scan and write files into
--forcefalseOverwrite existing raiflow.yaml and workflow

raiflow check

Runs compliance checks for a pipeline stage. By default opens a live browser dashboard; use --no-dashboard for headless/CI mode.

raiflow check --stage ci
raiflow check --stage ci --no-dashboard
raiflow check --stage ci --target app.py --no-dashboard
raiflow check --stage ci --threshold 0.7 --no-dashboard
raiflow check --stage ci --enable-llm-checks --no-dashboard
raiflow check --stage ci --dry-run --no-dashboard

Options

FlagDefaultDescription
--stagecipre-commit | ci | pre-deploy | post-deploy
--manifestraiflow.yamlPath to your compliance manifest
--outputraiflow-report.jsonPath to write the JSON compliance report
--target—Source file(s) to scan for Article 14 override endpoints (repeatable)
--threshold—Override per-rule pass threshold for all checks (0.0–1.0)
--enable-llm-checksfalseEnable semantic LLM-based evaluation (requires Ollama or API key)
--dry-runfalseRun all checks but always exit 0
--no-dashboardfalseHeadless mode — terminal output only
--dashboard-port8000Port for the live dashboard server

raiflow generate-tests

Reads the bundled eu_ai_act.yaml policy and generates a pytest file for every rule, so you can run compliance checks as part of your normal test suite.

raiflow generate-tests
raiflow generate-tests --output-dir tests/compliance

Pipeline Stages

Each stage runs a different subset of checks. pre-commit is lightweight and fast. ci and pre-deploy run the full suite including Article 11.

StageChecksTypical use
pre-commitBanned Model ScanFast local check before every commit
ciAll 8 checksPull request gate
pre-deployAll 8 checksPre-production gate
post-deploy5 checks (no Technical Docs / Bias / Robustness)Production monitoring

LLM Evaluation (--enable-llm-checks)

When --enable-llm-checks is passed, four checks gain a semantic evaluation path that reads document content and scores it against EU AI Act criteria using an LLM evaluator. The score (0.0–1.0) is compared against a per-rule threshold.

CheckEvaluatorThresholdScores on
Transparencytransparency_by_design_check0.85AI identification clarity, disclosure prominence, non-technical understandability
Risk Managementrisk_management_system_check0.85Risk completeness, mitigation depth, lifecycle coverage, vulnerable groups
Human Oversightintervention_capability_check0.90Intervention mechanisms, override clarity, operator comprehensibility
Technical Docstechnical_documentation_check0.90All five Annex IV elements: system description, training data, metrics, limitations, oversight
Fallback: If the LLM call fails for any reason, the check automatically falls back to the static result and appends (LLM unavailable — static fallback used) to the remediation hint. Your pipeline never breaks due to an LLM outage.

Supported backends

  • Local (Ollama): gemma2:2b, gemma2:9b, llama3.2:3b, mistral:7b — configured by raiflow init
  • API: Google Gemini 1.5 Flash/Pro, OpenAI GPT-4o/mini — set GEMMA_API_KEY or OPENAI_API_KEY

GitHub Actions Integration

raiflow init drops a ready-to-use workflow at .github/workflows/rai-compliance.yml with four jobs:

JobStageBlocks on failure
pre-commit-checkspre-commitPR merge
compliance-gateciBuild
build-and-sign—Signed artifact manifest
deploy-gatepre-deployDeployment
rai-compliance.yml (excerpt)
- name: Run compliance gate
  run: raiflow check --stage ci --no-dashboard
  env:
    RAIFLOW_MOCK_LLM: "true"   # mock LLM in CI if no API key

# With semantic checks (add your API key as a repo secret)
- name: Run semantic compliance gate
  run: raiflow check --stage ci --enable-llm-checks --no-dashboard
  env:
    GEMMA_API_KEY: ${{ secrets.GEMMA_API_KEY }}
CI auto-detection: In CI environments (CI=true or no TTY), the browser dashboard is automatically suppressed — no extra flags needed. The --no-dashboard flag is explicit but redundant in CI.

Live Dashboard

Running raiflow check --stage ci (without --no-dashboard) opens a browser dashboard at http://127.0.0.1:8000/.

  • Select pipeline stage and click Run Checks to trigger a live run
  • Results stream in real-time via Server-Sent Events (SSE)
  • Per-article cards with pass/fail badges, scores, and remediation hints
  • Click any card for detailed regulatory context and Annex IV citations
  • Overall compliance score and risk level (Low / Medium / High / Critical)
  • Download Report exports raiflow-report-<timestamp>.json

Compliance Report

Every run writes a JSON report to raiflow-report.json (configurable via --output). The report schema is stable across versions.

raiflow-report.json
{
  "schema_version": "1.0",
  "generated_at": "2026-04-17T12:00:00+00:00",
  "git_sha": "abc1234",
  "stage": "ci",
  "system_name": "my-rag-app",
  "overall_status": "pass",
  "evaluation_mode": "static",  // or "semantic"
  "checks": [
    {
      "article_id": "Article 13",
      "rule_id": "ART13-1",
      "check_name": "Transparency by Design",
      "status": "pass",
      "score": 1.0,
      "threshold": 1.0,
      "remediation_hint": ""
    }
  ]
}

The evaluation_mode field is "semantic" when --enable-llm-checks was used, "static" otherwise — useful for audit trails.
